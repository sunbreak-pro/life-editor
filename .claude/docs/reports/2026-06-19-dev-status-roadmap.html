3あなたの優先順位に沿ったロードマップ

ご要望（①デスクトップを先に完成して日常使い ②スマホ同期 ③タスク整理・スケジュール管理をストレスなく）を、最短で「毎日使える」状態にする順番に並べ替えました。

STEP 1 ｜ デスクトップを「日常使いできる」最小ラインへ 最優先・進行中

目安: 1〜2 週 ／ Phase 3 残り ＋ Phase 5-A の一部

殻（#79）は着地済み。「起動 → ログイン → タスク/スケジュールを毎日触れる」を本物にする工程。ここが終われば、まず自分の Mac で日常運用を開始できます。

  • macOS golden path 実機検証 — 起動 → ログイン → 各セクション操作が通ることを実機で確認
  • 未署名 .dmg をビルドして自分の Mac にインストール（右クリック→開く、$0）
  • Tray（常駐）＋ 自動起動 — 「閉じても常駐」「ログイン時に立ち上がる」= 日常ツールの必須要件
  • Windows / Linux は当面 CI ビルド成否のみ確認（友達配布は完成後）

到達点: 自分の Mac で Life Editor を毎日起動してタスク・スケジュールを運用できる。

STEP 2 ｜ タスク整理・スケジュール管理を「ストレスなく」詰める 並行可

目安: 1〜2 週 ／ パリティ W7（進行中）＋ W8 ＋ 実機目視 fix

毎日使ううえで体感に直結する操作感の仕上げ。STEP 1 と並行して進められます。

  • Tasks 詳細パネル（W7）を実装 — 一覧＋詳細の 3 ペイン。タイトル/状態/本文をその場編集（計画書 PR #92 / 実装は別セッション）
  • Schedule カレンダー充実（W8） — 週ビュー・ドラッグ操作など、スケジュール管理の密度を Desktop 同等へ
  • W1〜W6 の実機目視 fix — テーマ/フォント/i18n/Trash/コマンドパレット/Pomodoro/Audio/Analytics/Connect の体感確認と微修正（INDEX に 👀 多数）

到達点: タスクの並べ替え・詳細編集・予定の見通しがストレスなくできる。

STEP 3 ｜ スマホ同期 — Mobile の殻を起動可能にする 次の山

目安: 2〜4 週 ／ Phase 4

前述の通り同期ロジックの新規開発は不要。Mobile の殻を起動できれば Supabase 経由で Desktop と自動同期します。残るのは Capacitor の配線と動作確認。

  • Provider 配線 — Theme / Timer 等を mobile ルートに接続（scaffold では isNativeMobile() アダプタのみ提供で配線は申し送り）
  • iOS シミュレータ起動 → 無料 Apple ID + 7 日署名で自分の iPhone 実機検証（$0・友達配布は完成後）
  • Android AVD 起動 ＋ 未署名 .apk を自分の端末へ
  • safe-area / スプラッシュ / アイコン整備、Mobile 省略 Provider（Audio 等）の Optional 確認

到達点: 外出先でスマホからタスク・予定を確認/追記 → 帰宅後 Desktop に反映、の往復が成立。

STEP 4 ｜ 完成仕上げ（Phase 5-B / 5-C） 完成へ

目安: 2〜3 週

日常運用が回り始めた後の仕上げ。急がず進めてよい領域。

  • terminal-division 連携 — mcp-server を Postgres 版へ書き換え、ターミナルから AI が同一 DB を操作
  • 自動更新（electron-updater + GitHub Releases）／ Web URL 公開（Cloudflare Pages・$0）
  • 旧スタック削除（frontend/ src-tauri/ cloud/）＋ CLAUDE.md / vision / db-conventions を新スタック前提へ全面改訂

4データ同期はどう成立しているか

「スマホとデスクトップで同じタスク・予定が見える」を支える土台。すでに動く部分と、残る部分を分けて整理します。

✅ すでにある（再利用するだけ）
  • 単一データソース — 全端末が同じ Supabase(Postgres) を参照
  • DataService 境界 — UI は getDataService() 経由のみ。端末差を吸収
  • Realtime 購読（S8 / #47） — 他端末の変更が自動反映
  • LWW 同期モデル — items_meta.updated_at を基準に最終更新優先で衝突解決
  • Auth + RLS — 本人のデータのみアクセス（auth.uid() = user_id）
  • Timer/Audio テーブル（migration 0018）も同期対象
🔧 同期のために残ること
  • Mobile の殻を起動状態にする（STEP 3）— これが事実上の唯一の残作業
  • 各端末で同一アカウントでログインする導線確認
  • 常時オンライン前提の設計 — 圏外時はバナー表示（オフライン編集は Non-goal）
  • 実機での同期往復テスト（Desktop ↔ iPhone で同じタスクが見えるか）
注意（割り切り）： 本設計は オフライン編集を行いません（常時オンライン前提・Service Worker 不採用）。 電車内など圏外では「オンライン時にご利用ください」のバナーが出ます。N=1・毎日触る運用では実害が小さいという判断です。
