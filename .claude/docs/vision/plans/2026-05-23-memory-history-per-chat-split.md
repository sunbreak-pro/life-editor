## 0. このシステムが解決する問題（背景と本質）

**結論: comm/Outbox で確立済みの「単一書き込み者」原則を、MEMORY.md / HISTORY.md にも適用する。** これにより並行チャット運用での物理的な書き込み衝突をゼロにする。

### 現状の問題

`.claude/MEMORY.md` と `.claude/HISTORY.md` は **task-tracker スキル経由で複数チャットが同じ 1 ファイルに追記**する設計になっている。これは comm/Outbox の中核原則「**1 ファイル 1 チャット専用**」と真逆の構造で、以下の事故が実際に起きている（MEMORY 既存エントリより）:

- 並行チャット専有時の task-tracker オーバーライド ルール発生（共有 tracker 編集禁止・git add -A 禁止・pathspec commit）
- 末尾追記の同時実行で git merge conflict 多発
- 別チャットが計画書を自レーン SSOT に逃がす運用が常態化（= MEMORY/HISTORY が信頼できなくなった）

これは「3 人が同じノートの末尾に同時に書こうとして揉める」状態で、Git のマージでは解けない（後付けの調停は人手必須）。

### 本質的な解

冷蔵庫の付箋でたとえると、**家族で 1 枚の付箋を奪い合うのをやめ、各自が自分の付箋を貼り、ボードを見る人が全員分まとめて見る**ようにする。

```
現状:                           変更後:
.claude/                        .claude/
├── MEMORY.md (全員追記) ←衝突 ├── memory/
└── HISTORY.md (全員追記) ←衝突 │   ├── chat-engineer.md  (engineer のみ追記)
                                │   ├── chat-pm.md         (pm のみ追記)
                                │   ├── chat-qa.md         (qa のみ追記)
                                │   └── INDEX.md           (集約ビュー、自動生成)
                                └── history/
                                    ├── chat-engineer.md
                                    ├── chat-pm.md
                                    └── INDEX.md
```

これは comm/Outbox の構造をそのまま MEMORY/HISTORY に持ち込むだけで、概念は既に運用済みのものを再利用するため学習コストが低い。

### やらないこと（スコープ外）

- task-tracker スキルの責務拡張。**書き込みは従来どおり task-tracker 経由のみ、対象ファイルが per-chat に変わるだけ**
- MEMORY/HISTORY のスキーマ変更（見出しフォーマット・追記ルールは現行を踏襲）
- 自動アーカイブ・自動圧縮（現行 HISTORY-archive.md 運用は維持）
- リアルタイム集約（INDEX.md は task-tracker 実行時にだけ再生成。watcher は別計画 = FileChanged 計画の責務）

## 1. 設計の全体像

部品は 3 つ。**書き込み層（per-chat）/ 読み取り層（INDEX 集約） / Claude 起動時ロード層（CLAUDE.md auto-load）** で分解する。

```
[書き込み側]  各チャットの task-tracker が
              .claude/memory/chat-<name>.md に追記（単一書き込み者）
      │
      ▼
[集約生成]    task-tracker 実行時に
              memory/INDEX.md を全 chat-*.md から再生成
      │
      ▼
[起動時ロード] CLAUDE.md auto-load が
              memory/INDEX.md を context へ取り込む
              （ファイル名は CLAUDE.md 既存 auto-load 機構に合わせる）
```

### 自己判定（誰が書くか）

comm/Outbox と同じく、**チャット名はユーザーが宣言する**。task-tracker は `.claude/comm/.session-name` を読んで、自分が書くファイルを `chat-<name>.md` に固定する。

これは FileChanged 計画書 §1 の自己判定機構と完全に共通。**`.session-name` ファイルは両計画で同じ 1 つを共有**する（=どちらかの計画で先に作ったらもう片方はそれを使う）。

### INDEX.md の中身

INDEX.md は **読み手向けの集約ビュー**であって SSOT ではない。SSOT は各 `chat-*.md`。

```markdown
# MEMORY INDEX (auto-generated by task-tracker)

最終更新: 2026-05-23 14:30:00 (chat-engineer)

## 進行中タスク (across all chats)

- [chat-engineer] Data Unification DU-B: Tasks unification (in progress, plan: ...)
- [chat-pm] migration plan v3 review (waiting on QA)
- [chat-qa] Phase 3 plan QA findings (REVISE-REQUIRED)

## 直近完了 (across all chats)

- [chat-engineer] DU-A complete (apply success + verification) — 2026-05-23
- [chat-pm] Data Unification 親計画書 v3 — 2026-05-22
```

集約規則は task-tracker スキルに記述する（CLAUDE.md には書かない）。

## 2. FileChanged 計画との関係（重要）

### 共有する設計要素

両計画が**同じ部品**を必要とする。**先に進む側がインフラを敷き、後発側はそれを使うだけ**。

| 共有部品                       | FileChanged 計画での用途                                | per-chat 計画での用途                               |
| ------------------------------ | ------------------------------------------------------- | --------------------------------------------------- |
| `.session-name` ファイル       | 自己宛 Outbox を無視する判定                            | 自分が追記する `chat-<name>.md` を確定              |
| メイン作業ツリー正本パスの解決 | worktree から正本 Outbox を絶対パス監視                 | worktree から正本 memory/history を絶対パス読み書き |
| `.gitignore` 拡張              | `.session-name` / `.watch-state/` / `.inbox-pending.md` | 同じ `.session-name` を共有（追加不要）             |

### 補完関係（両方やると効果が乗算する）

- per-chat 分割 = **書き込み衝突を構造で消す**（物理レイヤー）
- FileChanged 監視 = **書き込みを相手に気付かせる**（通知レイヤー）

片方だけでも機能するが、**両方揃うと「並行チャットが互いの状態をリアルタイムで把握し、衝突せず働ける」**になる。これが本来の到達点。

### 推奨進行順序

**per-chat（本計画）を先**。理由:

1. 並行チャット運用を**構造的に安全**にしてから通知レイヤーを足すほうが、土台→上物の順で素直
2. FileChanged 計画 Phase 0 の `additionalContext` 注入検証が**未確認の不確実性**を抱えている（最悪 SessionStart cat の副系に劣化）。本計画はそういう未知数がない
3. 本計画は完了すれば**それ単体で MEMORY conflict をゼロに**できる。FileChanged を待たずに価値が出る

逆順（FileChanged 先行）にすると、書き込み層に衝突が残ったまま通知だけ早くなり、「気付いたが触れない（衝突回避でブロック）」が増えて運用が窮屈になる。

## 残課題・リスク

- **task-tracker スキル改修の影響範囲**: グローバルスキルなので、他プロジェクトの task-tracker 動作にも影響する。Phase 5 までは「`.claude/memory/` ディレクトリが存在するときだけ per-chat、無ければ従来通り MEMORY.md」のフォールバック実装が必要
- **CLAUDE.md auto-load の互換性**: CLAUDE.md 内の MEMORY.md 直参照が他にもあるかもしれない。Phase 0 で grep を徹底
- **既存 outbox との二重管理感**: `.claude/comm/outbox/chat-*.md` と `.claude/memory/chat-*.md` が並ぶ構造になる。役割分担を明確化する（comm = 他チャット宛のメッセージ / memory = 自分のタスク状態）。混同したら設計が壊れる
- **チャット名宣言の取りこぼし**: `.session-name` 未設定時に task-tracker が起動した場合の挙動を決める（エラーで止めるか、デフォルト `chat-main` で進めるか）。推奨はエラー停止（事故防止）
- **INDEX.md の鮮度**: 集約は task-tracker 実行時のみ。最後の更新から時間が経った INDEX.md を読むと古い可能性。**FileChanged 計画が完成すれば、他チャットの memory 更新を即時通知できるので INDEX 鮮度問題は自然解消** = 両計画が補完する根拠
- **worktree 跨ぎ書き込みのアトミック性**: 案 A で各 worktree が正本 memory に絶対パス追記する場合、append 自体は OS のファイルロックで安全だが、INDEX.md 再生成のレースは別途考慮（最後に書いた人が勝つ＝LWW で良いか要確認）
- **archive 規約の決定先送り**: per-chat 化により `chat-*.md` の数が増える。各ファイルのアーカイブ閾値・周期を Phase 2 で決める
