# Plan: Claude Desktop 風チャット UI（CLI 出力から構成・サブスクリプション利用前提）

> 本書は **コンセプト計画書** である。具体的なファイル配置・関数シグネチャ・ライブラリ選定は意図的に書かない。「何を / なぜ / どういう原理で」だけを定める。実装詳細は本計画が承認された後、別ドキュメントに分割する。

---

## 1. Context

### 1.1 動機

life-editor は既にアプリ内 Terminal で Claude Code を起動し、MCP 経由で全データを自然言語操作できる（CLAUDE.md §5.2）。しかし体験は **「黒い画面に文字が流れる開発者向け CLI」** にとどまっている。

一方、Claude Desktop は **「会話の単位で吹き出しが積み上がり、ツール呼び出しは折り畳みカードで、コードはシンタックスハイライト付きのコードブロックで提示される」** という、非エンジニアでも扱える対話 UI を提供している。

life-editor の Vision「AI と会話しながら生活を設計・記録・運用するパーソナル OS」を完成させるには、**「ターミナル」ではなく「対話画面」としての UI** が必要になる。

### 1.2 制約 — サブスクリプション縛り

- CLAUDE.md §1 Non-Goals に「Claude API 直課金」が明記されている
- 課金チャネルはユーザーが既に契約している **Claude のサブスクリプション**（Pro / Max 等）に限る
- API キーをアプリに保存させない（鍵管理 UX を持たない、漏洩リスクを背負わない）
- 結論: **「ユーザーのマシン上に既にあるサブスク認証済み Claude を子プロセスとして起動する」** が唯一の現実解

### 1.3 Non-goals

- Claude Desktop の UI を完全再現すること（全コンポーネント一致は目指さない）
- マルチモデル対応（GPT / Gemini 切替 UI）
- ブラウザ上での動作（life-editor は Web UI を Vision で否定済み）
- 複数アカウントの並列管理
- Mobile 対応（CLAUDE.md §2 通り、本機能は Desktop 専用）

## 2. コアコンセプト

本計画を貫く 4 つの原理。

### 2.1 Subscription-First

> ユーザーが既に課金しているチャネルを再利用する。アプリは課金主体にならない。

- 鍵を預からない／鍵を要求しない
- ユーザーが事前に CLI 側で認証を済ませている前提でアプリが立ち上がる
- アプリ側に必要なのは「認証済みかどうかを観測する手段」と「未認証ならガイドする UI」だけ

### 2.2 CLI-as-Backend

> CLI は単なる人間向けの黒画面ではなく、**構造化された AI ランタイムへのプロセス境界** として扱う。

- アプリの責務は「対話モデルを動かすこと」ではなく「動いている対話モデルの入出力を翻訳すること」
- モデル・ツール実行・会話継続・MCP 接続はすべて CLI 側に委譲
- CLI が壊れても life-editor 本体は壊れない（疎結合）

### 2.3 構造化ストリーム優先（ANSI ではなく Event）

> ターミナルの ANSI バイト列を後付けでパースして UI に直すのは泥沼。最初から構造化された出力を要求する。

- 「テキスト」「ツール呼び出し」「ツール結果」「終了マーカー」が **別個のイベント** として届く前提に立つ
- イベント単位で UI コンポーネントが対応する（吹き出し / カード / バナー）
- ANSI 路線は **既存 Terminal タブの責務** として温存し、本機能では使わない（責務分離）

### 2.4 既存資産の再利用

> life-editor は Terminal / MCP / Theme / DataService を既に持っている。チャット UI はそれらの **再構成** であって新規構築ではない。

- プロセス起動・入出力配管 → 既存 PTY 路線の知見が転用できる
- データ操作の手足 → 既存 MCP Server がそのまま使える（30 ツール）
- 配色・フォント・ダーク/ライト → 既存 `notion-*` トークンに従う
- 設定・履歴の保存先 → 既存 SQLite (CLAUDE.md §4) の延長で扱える

## 3. ユーザー体験のコンセプト

実装ではなく「ユーザーから見て何が起きているか」を定義する。

### 3.1 起動と認証の体験

- アプリ起動時、サブスク認証の状態が **静かに観測される**
- 未認証なら、チャットセクションは「外部 CLI でログインが必要です」というガイド画面のみを出す（鍵入力フォームは持たない）
- 認証済みなら、即座に空のチャット画面が立ち上がる

### 3.2 会話の単位

- 「セッション」という概念をユーザーに見せる（左ペインに会話履歴一覧）
- 1 セッション = 1 つの連続した文脈
- 「新しい会話」「過去の会話を再開」「会話を分岐」がトップレベル操作
- セッションの実体は CLI 側が管理しているものを **参照しているだけ**（life-editor は ID を覚えるのみ）

### 3.3 メッセージの種別

ユーザーから見える情報単位は次の 4 種類だけに整理する：

| 種別            | UI 表現                                  | 目的                                |
| --------------- | ---------------------------------------- | ----------------------------------- |
| ユーザー発話    | 右寄せ吹き出し                           | 自分が言ったことの再確認            |
| AI 応答テキスト | 左寄せ吹き出し（Markdown 描画）          | 通常の返答                          |
| ツール呼び出し  | 折り畳みカード（ツール名 + 引数 + 状態） | 「AI が何をしようとしたか」の可視化 |
| ツール結果      | カードの下段（成功 / 失敗 + 抜粋）       | 「実際に何が起きたか」の可視化      |

> Claude Desktop の最大の発明はこの **「ツール呼び出しを折り畳みカードに分離した」** こと。ターミナルでは混ざる情報が、この分離により非エンジニアでも追える。

### 3.4 ストリーミングの体感

- AI 応答は逐次到着するため、**到着途中はカーソル点滅 or プログレスインジケータ**で「考え中」を表現
- 確定したテキストは Markdown として再レンダリング（コードブロック・テーブル・リスト）
- ツール実行中は対応カードが「実行中」状態 → 結果到着で「完了」状態に遷移

### 3.5 中断と復帰

- 送信中は「停止」操作が常に可能（CLI プロセスへの中断シグナル）
- アプリを閉じても会話は CLI 側に残っているので、再開時にセッション ID から復元できる
- life-editor 側で完結したい誘惑に抗うこと（CLI を SSOT にする）

### 3.6 life-editor との統合体験

- 「今日のタスクを要約して」「来週の Schedule を組んで」など、**life-editor のデータを前提にした指示が自然に通る**（既存 MCP の延長）
- 逆向きの統合：チャットの結果から「このタスクを Tasks セクションで開く」のような **deep link** が踏めると、UI 統合の意味が出る
- どこからでも呼び出せる「グローバル質問入力」を持つかどうかは Phase の後段で判断

## 4. 構成要素のコンセプト（5 層モデル）

実装単位ではなく **責務の層** として定義する。

```
[ User Input UI ]                                ← 入力欄・送信・停止・添付
       ↓
[ Conversation State ]                           ← セッション一覧・現在セッション・メッセージ列
       ↓
[ Event Translator ]                             ← 構造化イベント ⇄ UI モデルの双方向変換
       ↓
[ Process Bridge ]                               ← 子プロセス起動・stdin/stdout・lifecycle
       ↓
[ External CLI Runtime (subscription auth) ]     ← life-editor の責務外
```

### 4.1 Process Bridge

> 「子プロセスを生かし続けるレイヤー」。

- 起動・終了・入力送出・出力受信・異常検知・再起動だけを担当
- ここは UI を一切知らない（テストしやすさのため）
- 既存の Terminal 実装（PTY 路線）と **隣接するが別系統**。同じプロセスを 2 系統から触らない

### 4.2 Event Translator

> 「CLI が吐く構造化イベント」⇔「UI が描く Conversation Model」を翻訳する層。

- 1 行 1 イベントの正規化
- 「ストリーム途中の text delta」を「UI 上の確定 message」に集約する責務
- ツール呼び出し/結果の対応付け（呼び出し ID で紐づける）
- 翻訳に失敗したイベントは **黙って捨てない**（デバッグログ送り）

### 4.3 Conversation State

> 「いま何を表示すべきか」の唯一の情報源。

- セッション一覧、各セッションのメッセージ列、送信中フラグ、停止可否
- ここは React の State 管理レイヤとして既存 Pattern A (Context/Provider) に従う
- 永続化は最小限（セッション ID と送信履歴の参照のみ）。本文は CLI 側に問い合わせ可能

### 4.4 User Input UI

> 「ユーザーが言葉を渡す唯一の窓口」。

- IME 安全（既存規約 §6.6 — `isComposing` チェック）
- 送信中は「送信」が「停止」に変わる
- TipTap か textarea かはデザイン段階の議論（コンセプトでは決めない）

### 4.5 Message Rendering

> 「届いた言葉を読み物として整える層」。

- Markdown / コード / テーブル / リスト / 数式（必要なら）
- ツールカードのレイアウト
- 既存 `notion-*` トークン準拠（不透明背景の規約 §6.4 を必ず守る — 透明落ち禁止）

## 5. データフローの概念図

```
[User types message]
        │
        ▼
[Conversation State: append user message, mark "in flight"]
        │
        ▼
[Process Bridge: feed prompt + session id to CLI stdin]
        │
        ▼  (1 行 1 イベントのストリーム)
[Event Translator]
   ├─ "assistant text delta"   → 現在進行中の吹き出しに追記
   ├─ "tool use start"          → 新しいツールカードを "running" で追加
   ├─ "tool use result"         → 対応カードを "done/failed" に更新
   └─ "result / session id"     → セッション ID を保存、in-flight 解除
        │
        ▼
[Conversation State updates → React re-render]
        │
        ▼
[User reads / interrupts / replies]
```

要点：

- イベントは **追記モデル**（過去の確定要素を書き換えない）
- 「考え中」の状態は in-flight フラグ 1 つで表現
- 失敗パス（プロセス死亡 / 認証失効 / タイムアウト）も **同じイベント形式**で UI に届ける（特例分岐を増やさない）

## 6. 既存資産との関係

| 既存資産                             | 本計画での扱い                                   | 衝突回避                                         |
| ------------------------------------ | ------------------------------------------------ | ------------------------------------------------ |
| Terminal Panel (xterm.js + PTY)      | 残置。「素のターミナル」が必要な場面用           | チャット UI は別セクション or 別パネルとして併存 |
| MCP Server (30 tools)                | そのまま CLI 側に接続される                      | アプリは MCP 設定の引き渡しだけ気にする          |
| DataService 抽象                     | チャットの保存物（セッション ID 等）はここを通す | CLI 直叩きしない、IPC 経由                       |
| Theme / `notion-*` トークン          | チャット UI も準拠                               | 透明背景禁止規約 §6.4 を最初から守る             |
| i18n (en / ja)                       | チャット枠の固定文言は対応                       | AI の応答自体は AI が言語を判断                  |
| Provider 順序 (§6.2)                 | 新 ChatProvider を末尾に追加（外側依存のみ）     | 他 Provider に依存させない設計を維持             |
| ScheduleItems / Tasks 等の deep link | 「結果から開く」リンクの受け先                   | 双方向リンクは Phase 後段                        |

> **Mobile**: 本機能は Desktop 専用（既存 Terminal と同じ理由）。Mobile 省略 Provider と同じ扱いで、Mobile ビルドからは Provider ごと外す。

## 7. リスクとトレードオフ

| リスク                                  | 性質           | 緩和方針                                                                                         |
| --------------------------------------- | -------------- | ------------------------------------------------------------------------------------------------ |
| CLI の出力フォーマット破壊変更          | 外部依存リスク | バージョン固定 or 起動時に CLI バージョン検出 → 非対応なら明示的に劣化動作                       |
| 認証失効 / サブスク失効                 | 体験リスク     | エラーイベント → ガイド画面に **戻す**（鍵入力 UI は出さない）                                   |
| プロセス起動コスト                      | 体感リスク     | セッションごとに長寿プロセス、毎メッセージ起動はしない                                           |
| 大量出力でのレンダリング負荷            | 性能リスク     | 仮想スクロール / コードブロックの遅延ハイライト                                                  |
| ANSI 出力との混入                       | 設計リスク     | 構造化ストリーム専用モードに限定し、TUI モードは Terminal タブに隔離                             |
| ユーザーが CLI の存在を意識せず壊す     | UX リスク      | CLI のインストール状態を起動時に観測し、未導入なら導入ガイドを出す                               |
| 友達向け配布（Windows/Android）との関係 | スコープリスク | **本機能は作者向け / Desktop のみ**。配布ビルドでは feature flag で隠す（Cloud Sync と同じ運用） |

## 8. 段階的アプローチ（Phase 区切り、概念のみ）

> 各 Phase は **観測可能な体験差分** で区切る。実装タスクの羅列ではない。

### Phase 0 — 認証観測と空殻

> 「鍵を預からない」を最初に実装ポリシーとして固定する。

- アプリは CLI が認証済みかを観測できる
- チャット用セクションが空殻として存在する（未認証ガイドのみ）
- これだけで「サブスク縛り設計が成立するか」が検証できる

### Phase 1 — 一往復のチャット

- ユーザー入力 → CLI 一発実行 → テキスト応答が吹き出しに表示される
- ツール呼び出しはまだ素のテキストとして混ざっていてよい
- ストリーミング表現は最小（「考え中…」だけ）
- ここまでで「概念モデルが現実に通るか」が決まる

### Phase 2 — 構造化イベントと折り畳みカード

- 出力を構造化ストリームに切り替える
- ツール呼び出し / 結果が独立カードになる
- Markdown レンダリング・コードシンタックスハイライト
- ここで初めて「Claude Desktop 風」が体感できる

### Phase 3 — 会話の永続化と分岐

- セッション一覧 / 再開 / 名前付け
- 「過去会話を続ける」「分岐させる」
- セッションメタデータの保存場所を SQLite に確保（本文は CLI 任せ）

### Phase 4 — life-editor 統合

- 結果メッセージから既存セクション (Tasks / Schedule / Notes) への deep link
- 入力欄に「現在開いているエンティティをコンテキストに含める」スイッチ
- ここで初めて「パーソナル OS の AI 窓口」になる

### Phase 5（任意） — グローバル呼び出し

- どの画面からでも呼び出せる Cmd+K 的入力
- 当面は見送り。Phase 4 までの体感が十分なら Phase 5 は不要かもしれない

## 10. オープン論点（Phase 着手前に決める）

これらは本書では結論を出さない。実装着手時に判断する。

- **配置**: 新セクション (`chat`) として SectionId を増やすか、既存 Terminal Panel と並ぶ第二パネルにするか
- **入力 UI の実装**: TipTap か素の textarea か（複数行 / 添付の必要性に依存）
- **セッション一覧の永続化粒度**: アプリ側 DB に最小メタだけ持つか、本文の検索インデックスまで持つか
- **deep link の方向**: 「チャット → エンティティ」だけか、「エンティティ → そのコンテキストでチャット」も用意するか
- **CLI 未導入ユーザーへの導線**: ガイド表示のみか、自動セットアップウィザードまで踏み込むか
- **複数 OS での CLI 検出**: macOS / Windows で実行ファイル探索戦略をどこまで自動化するか

## 11. 成功条件

本機能が成功したと言える観測可能な状態：

1. ユーザーが API キーを **一度もアプリに入力せずに** AI 対話ができている
2. ツール呼び出しが折り畳みカードで提示され、エンジニアでない閲覧者でも何が起きたか追える
3. 既存 MCP 経由で life-editor のデータが対話から操作できている（Phase 4）
4. CLI 側のフォーマット変更で UI が**完全には壊れず**、最低でもテキスト応答だけは表示される（劣化耐性）
5. Mobile ビルド・友達配布ビルドに **本機能が混入していない**（feature flag 健全性）

## 13. まとめ

この計画書は「**life-editor の対話体験を、CLI を背中に隠した Claude Desktop 風 UI に置き換える**」というコンセプト宣言である。

- 課金は既存サブスクに乗る（鍵を預からない）
- CLI を AI ランタイムとして扱う（自前で LLM を動かさない）
- ANSI ではなく構造化イベントを描く（ターミナルではなくアプリにする）
- 既存資産（MCP / Theme / DataService）を再構成する（新規構築ではない）

実装に進む際は、本書の §8 Phase 区切りを外形的なマイルストーンとして使い、各 Phase の終わりに「観測可能な体験差分」が成立しているかをレビューする。実装詳細・ファイル配置・関数設計は、Phase 0 着手の直前に別ドキュメントへ展開する。
