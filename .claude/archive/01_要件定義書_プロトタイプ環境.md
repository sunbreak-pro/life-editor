## 1. 背景と目的

### 1.1 背景

現状、モバイルUIの開発は本番コードベース(`frontend/`)上で直接行われており、以下の課題がある:

- UI/UXのトライ&エラーがTauri/Electronビルドのオーバーヘッドに引きずられる
- データ層(Supabase / SQLite / DataService)の挙動と絡まり、純粋なUIの正解が見えにくい
- ホットリロード再起動や型エラー解消に時間を取られ、デザイン上の意思決定が遅延する
- Claude Artifacts上で3つのデモTSXを磨いてきたが、**実環境での挙動確認(ルーティング、画面遷移、複数画面の統合)** が出来ない

### 1.2 目的

**UI/UX工程とCRUD工程を完全に分離**し、UI/UX工程専用の軽量プロトタイプ環境を構築する。

1. **意思決定の高速化**: ランタイム依存を取り除き、Vite単体で1秒未満のホットリロードを実現
2. **段階的な統合**: 「独立画面の磨き」→「3画面統合テスト」→「CRUDモック化」の3段階を同じ環境内で遷移可能にする
3. **将来移植性の確保**: Tauri にも Electron + Capacitor にも乗せられる **中立なUIレイヤー** として作る
4. **Claude Codeとの分業**: プロトタイプで完成したUIを Claude Code に「差分移植」させる前提のディレクトリ構造を整える

### 1.3 非目標(Non-Goals)

- 本番 `frontend/` のコードに手を入れない(独立並置)
- 本番のデータ層(Supabase / DataService / IPC)をプロトタイプで再現しない
- Tauri / Electron / Capacitor のいずれかへのバンドリングは行わない(純粋Webのみ)
- E2Eテスト・スナップショットテスト等の自動テスト整備は行わない
- iOS実機・実機ビルドへの転送は行わない(ブラウザ確認のみ)

### 2.2 本番との関係

## 3. 要件

### 3.1 機能要件

#### FR-1: 独立画面起動モード

3つの画面をそれぞれ独立したURLで起動できる。

- `/schedule` → ScheduleScreen のみを `max-w-md` のフレームで表示
- `/work` → WorkScreen のみ
- `/materials` → MaterialsScreen のみ
- 各画面は他画面の影響を一切受けない(状態・スタイル共に独立)

#### FR-2: 統合プレビューモード

3画面を本番想定の **下タブナビ** で切替可能な統合ビューを起動できる。

- `/unified` → 下タブで Schedule / Work / Materials / Settings を切替
- Settings はダミー(空画面 or 「未実装」表示)で可
- タブ切替は React state のみで実装(URLは `/unified` のまま)

#### FR-3: 開発者用インデックス画面

プロトタイプのルート `/` にアクセスすると、各モードへのリンク集を表示する。

```
- /schedule    (独立)
- /work        (独立)
- /materials   (独立)
- /unified     (統合プレビュー)
```

#### FR-4: モックインタラクション

CRUDが未実装の状態でも、ボタン・タブ・スワイプ等の **UIインタラクションだけは反応する**。

- ボタンクリック → `console.log()` でアクション名を出力(動作確認用)
- データ書込み系 → アラート or トースト「(モック)保存しました」表示でも可
- データ取得 → ファイル内のハードコードされたダミーデータを返す
- 状態変更が画面に反映される必要は**なし**(クリックが反応するだけでOK)

### 3.2 非機能要件

#### NFR-1: 起動速度

`npm run dev` から1秒以内にブラウザ表示可能。ホットリロードも1秒未満。

#### NFR-2: 依存最小化

本番が将来 Electron に移行することを見越し、以下の依存のみに絞る:

```
- react / react-dom
- react-router-dom        (例外的に許可)
- lucide-react
- vite / @vitejs/plugin-react
- typescript
- tailwindcss(v3) / postcss / autoprefixer
```

`@tauri-apps/*` / `electron` / `@capacitor/*` / Supabase / SQLite / MCP 関連は **一切入れない**。

#### NFR-3: ファイル独立性

3つの画面TSXは依然として **1ファイル完結** を維持する。プロトタイプ用に共通化や分割は行わない(本番移植時の差分を最小化するため)。

#### NFR-4: ブラウザ専一

動作確認は Chrome / Safari / Firefox の最新版のみ対象。モバイルブラウザ確認は任意(必須でない)。

### 3.3 制約事項

| 制約                                       | 理由                                                                      |
| ------------------------------------------ | ------------------------------------------------------------------------- |
| `localStorage` / `sessionStorage` 使用禁止 | 本番ルールと一致、Artifacts互換                                           |
| 1ファイルTSX構造を崩さない                 | 本番移植時の差分最小化                                                    |
| Catppuccin Mocha 配色固定                  | 本番デザイントークンと一致                                                |
| Provider / Context は使わない              | 本番Mobile版が Optional Provider 戦略のため、プロトタイプでは無くしておく |

## 4. アーキテクチャ方針

### 4.1 設計原則

**「Tauri にも Electron にも乗せられる中立なUIレイヤー」** を作る。

具体的には:

1. **ランタイム非依存**: `window.__TAURI__` / `window.electron` / Capacitor API を一切参照しない
2. **データ非依存**: 全データはファイル内ハードコード(モック)
3. **ルーティングは Web 標準**: `react-router-dom` の HashRouter or BrowserRouter

### 4.2 移植戦略

プロトタイプで完成したUIを本番(`frontend/src/components/Mobile/`)へ移植する際、Claude Code に以下を依頼する:

```
このプロトタイプの ScheduleScreen.tsx を、
本番の frontend/src/components/Mobile/MobileCalendarView.tsx と diff を取り、
ロジック層(useCalendarTags, DataService 等)を保持したまま、
プレゼンテーション層のみ置換してください。
```

そのため、プロトタイプ側のTSXは **プレゼンテーション層が明確に切り出された構造** を維持する(useEffect でのデータ取得等はモック関数で代用)。

## 7. データの扱い

### 7.1 ダミーデータ

各 Screen TSX 内で `const MOCK_DATA = [...]` のように **ファイル冒頭にハードコード** する。

例(ScheduleScreen.tsx 内):

```typescript
const MOCK_EVENTS = [
  { id: "1", title: "チーム会議", date: "2026-05-21", color: "mauve" },
  { id: "2", title: "デザインレビュー", date: "2026-05-22", color: "sky" },
];
```

### 7.2 モック関数

CRUD操作は console.log + 視覚的フィードバックのみ:

```typescript
const handleAddEvent = (e: Event) => {
  console.log("[MOCK] addEvent:", e);
  // 状態は更新しない or 一時的に local state へ append のみ
};
```

### 7.3 永続化なし

画面リロードでダミーデータは初期状態に戻る(これは仕様、バグではない)。

## 9. リスクと緩和策

| リスク                              | 影響度 | 確率 | 緩和策                                                            |
| ----------------------------------- | ------ | ---- | ----------------------------------------------------------------- |
| 本番との依存バージョン差            | 中     | 中   | プロトタイプは独立 `package.json` で固定、本番影響なし            |
| TSXの肥大化                         | 低     | 高   | 1ファイル完結を維持、コンポーネント分割はしない                   |
| Tailwind v4記法を誤って混入         | 中     | 中   | tailwind.config.js を v3 構文で書く + grep監査                    |
| Claude Codeへの移植が雑になる       | 中     | 中   | プレゼンテーション層を明確に切り出す + 移植時に diff レビュー必須 |
| react-router の例外許可が他に広がる | 低     | 低   | README に「react-router 以外の例外を作らない」と明記              |
